## 🚧 현재 블로킹 이슈

### Issue 1: bytes 타입 부재

**현상**:
```freelang
// 원하는 것 (불가능):
let elf_header = bytes([0x7f, 0x45, 0x4c, 0x46])

// 현재 가능한 것:
let elf_header = "[불가능]"
```

**영향**:
- ELF 헤더 작성 불가
- x86-64 기계어 코드 생성 불가
- 바이너리 상수 임베드 불가

**필요**: `bytes` 타입 추가 + 관련 builtin 함수

---

### Issue 2: 메모리 관리 부재

**현상**:
```freelang
// 원하는 것 (불가능):
let ptr = malloc(1024)
ptr[0] = 0x48  // 메모리 쓰기

// 현재: 불가능
```

**영향**:
- 동적 메모리 할당 불가
- 포인터 산술 불가
- 런타임 배열 구현 불가

**필요**: `malloc()`, `free()`, 포인터 읽기/쓰기

---

### Issue 3: 바이너리 연산 부족

**현상**:
```freelang
// x86-64 인코딩 예시:
// mov rdi, 0x400000   → 48 BF 00 00 40 00
// call rel32          → E8 <32비트 offset>

// 현재: 불가능 (복잡한 바이너리 조작)
```

**영향**:
- 동적 링크 정보 생성 불가
- ELF 코드 섹션 생성 불가
- 상대 주소 계산 어려움

**필요**: 비트 연산, 엔디안 변환, x86-64 인코더

## 📋 해결 계획 (우선순위)

### Level 1️⃣: bytes 타입 구현 (가장 중요)

**작업**:
1. `include/runtime.h`에 `fl_bytes_t` 구조체 추가
2. `src/stdlib.c`에 builtin 함수 구현:
   - `bytes_new(size)`
   - `bytes_len(bytes)`
   - `bytes_set(bytes, idx, val)`
   - `bytes_get(bytes, idx)`
   - `bytes_write_u32()`, `bytes_write_u64()`
3. VM에 FL_TYPE_BYTES 처리 추가

**예상 시간**: 2-3 시간
**예상 코드량**: 300-500 줄 (C)

---

### Level 2️⃣: 메모리 관리 구현

**작업**:
1. 간단한 힙 할당자 구현
2. `builtin_malloc()`, `builtin_free()` 추가
3. 포인터를 정수로 표현 (intptr_t)

**예상 시간**: 1-2 시간
**예상 코드량**: 200 줄 (C)

---

### Level 3️⃣: x86-64 인코더 (FreeLang)

**작업**:
1. `encode_mov_rdi_imm64()` - mov rdi, imm64
2. `encode_call_rel32()` - call rel32
3. `encode_x86_ret()` - ret
4. ELF 헤더 생성 함수

**예상 시간**: 1 시간
**예상 코드량**: 100-200 줄 (FreeLang)

## 🎓 교훈 & 인사이트

### 1. 배열 초기화 버그 (전역 vs 로컬)
- **문제**: 전역 `let arr = []`이 배열이 아닌 문자열로 초기화됨
- **해결**: 함수 내부에서만 배열 사용
- **교훈**: 스코프별 초기화 로직 검증 필요

### 2. FreeLang의 한계
- **강점**: Lexer, Parser는 완전히 Pure FreeLang으로 가능
- **약점**: 저수준 작업 (바이너리, 메모리)은 C 필요
- **결론**: 2-tier 언어 설계 (High-level + Low-level)

### 3. 자체호스팅의 정의
- **현재**: Lexer + Parser (Pure FreeLang) → ✅ 자체호스팅 부분 증명
- **추가 필요**: Code Generation (bytes + 메모리) → C 보조 필요
- **결론**: **하이브리드 자체호스팅** (FreeLang + C)
